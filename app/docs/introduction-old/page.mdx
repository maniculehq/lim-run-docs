# Introduction (old draft)

Limrun gives you remote, programmable access to the mobile development hardware that lives on a Mac: iOS simulators, an attached Xcode build sandbox, and Android emulators. Every instance is reachable over the REST control plane (`api.limrun.com`), per-instance HTTP and WebSocket endpoints, an ADB tunnel (Android), a per-instance MCP server, or the `lim` CLI. That means any host can drive it, including Linux CI runners and cloud coding agents that have no access to a Mac.

## What you can build

- **Cloud coding agents that ship iOS code.** Wire your agent's terminal to the `lim` CLI. The agent runs `lim ios create --xcode`, calls `lim xcode build .`, drives the simulator with `lim ios tap-element` / `lim ios screenshot`, and posts a preview link back to the PR. No `xcodebuild` on the agent's host, no macOS runner.
- **Embedded simulators in your product.** Render a live iOS or Android simulator inside your web app using `<RemoteControl />` from `@limrun/ui`. The browser streams over WebRTC; your server holds the API key. Used by platform integrators like coding-agent providers to give their users a real device on the page.
- **Mobile CI from a Linux box.** Run `xcodebuild`, Appium, Playwright Android, or Detox against a Limrun instance from any GitHub Actions runner. No macOS minutes, no `runs-on: macos-latest`.
- **No-Mac iOS development.** Build, run, and test iOS apps from a Windows laptop, a Chromebook, or a Linux desktop. Source code stays local; only the build runs remotely.

## The four resources

Limrun exposes four top-level resources. Each one is independent: you can use the iOS simulator without an Xcode sandbox, or stand up a standalone Xcode builder with no simulator attached.

| Resource | What it is | When you use it |
|---|---|---|
| **iOS instance** | A booted iOS Simulator (iPhone; iPad and Apple Watch in alpha), streamable, programmable, optionally with an Xcode build sandbox attached | Driving the UI, running an app, IAP testing, embedding in a web app |
| **Xcode instance** | A standalone Xcode build sandbox: source-sync + remote `xcodebuild` with no simulator | CI builds, real-device IPAs for distribution, MCP-wrapped build tools |
| **Android instance** | A booted Android emulator with an ADB tunnel | Anything ADB does locally, plus Appium, Playwright Android, scrcpy streaming |
| **Asset** | A managed binary store for `.ipa`, `.app.tar.gz`, `.apk`, and configuration files | Pre-installing apps at boot, hosting build artifacts, sharing previews |

The iOS and Xcode resources compose: an iOS instance can be created with `spec.sandbox.xcode.enabled` to get a paired Xcode sandbox in one call. The Xcode sandbox also runs as a top-level resource if you don't need a simulator at the same time.

## How it fits together

The CLI, SDKs, MCP, and `<RemoteControl />` are all clients of the same REST API at `https://api.limrun.com`. Your org API key (prefixed `lim_`, via `LIM_API_KEY`) authenticates the control plane and the per-instance MCP server. Each instance also returns a per-instance `status.token` for device control (`apiUrl`, `endpointWebSocketUrl`, ADB tunnel, and `<RemoteControl />`). You can mix and match: create an instance from a Node backend, stream it to a React frontend, and control it from a Python notebook on the side.
