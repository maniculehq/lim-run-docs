# Automatic PR Previews with GitHub Actions

An iOS PR preview works like a web deploy preview: every pull request gets its own build and its own link, and reviewers open it in a browser from anywhere. Limrun does the build, the asset upload, and the preview URL. GitHub Actions handles the workflow trigger and the PR comment.

This page walks through a GitHub Actions workflow built around the standard `lim` CLI commands. Every push to a PR builds the app on Limrun, uploads the artifact, and posts a preview link on the PR.

## What you get on the PR

After the workflow runs, the PR has a comment like this:

```
📱 iOS preview ready

  Preview: https://console.limrun.com/preview?asset=my-app-pr-42.zip&platform=ios
  Build:   1m 47s

Open the preview link on any device. The Limrun console will hand off to a remote simulator pre-installed with this build.
```

## Required permissions

The workflow needs to comment on the PR, so grant `pull-requests: write`:

```yaml
permissions:
  pull-requests: write
  contents: read
```

If your project lives in an organization with restrictive defaults, also confirm Actions are allowed to comment on PRs in the repo's Actions settings.

## Required secrets

Set these in **Settings → Secrets and variables → Actions** on the GitHub repo:

| Secret | Source |
|---|---|
| `LIM_API_KEY` | [console.limrun.com](https://console.limrun.com), API keys section |
| `P12_BASE64` | (Only if signing) `base64 -i dist.p12 \| pbcopy` |
| `P12_PASSWORD` | (Only if signing) PKCS#12 password |
| `PROFILE_BASE64` | (Only if signing) `base64 -i MyApp.mobileprovision` |

`GITHUB_TOKEN` is provided automatically by Actions; you don't need to create one.

## The workflow

The minimum end-to-end workflow. It builds on every PR open or sync, uploads with a PR-scoped asset name, and posts a comment with the preview link.

```yaml .github/workflows/ios-preview.yml
name: iOS Preview

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  pull-requests: write
  contents: read

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Limrun CLI
        run: npm install --global @limrun/cli

      - name: Build on Limrun
        env:
          LIM_API_KEY: ${{ secrets.LIM_API_KEY }}
          PR: ${{ github.event.number }}
          REPO: ${{ github.event.repository.name }}
        run: |
          ASSET_NAME="${REPO}-pr-${PR}.zip"
          echo "ASSET_NAME=${ASSET_NAME}" >> $GITHUB_ENV

          lim xcode create --reuse-if-exists \
            --label pr=${PR} \
            --label repo=${REPO}

          lim xcode build . \
            --scheme MyApp \
            --upload "${ASSET_NAME}"

      - name: Comment with preview link
        uses: actions/github-script@v7
        with:
          script: |
            const assetName = process.env.ASSET_NAME;
            const url = `https://console.limrun.com/preview?asset=${encodeURIComponent(assetName)}&platform=ios`;
            const body = [
              '📱 **iOS preview ready**',
              '',
              `Preview: ${url}`,
            ].join('\n');
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body,
            });
```

### What each step does

1. **Checkout** is standard. The source tree is what gets synced to the Xcode sandbox.
2. **Install Limrun CLI** pulls `@limrun/cli` from npm. The build runs on Limrun, so the job doesn't need a macOS runner.
3. **Build on Limrun** is the actual build. `lim xcode create --reuse-if-exists` provisions (or returns) an Xcode sandbox scoped to this PR via labels. Then `lim xcode build . --upload <name>` syncs the source, runs `xcodebuild`, and uploads the artifact to [Asset Storage](/docs/platform/asset-storage). Naming the asset `${repo}-pr-${PR}.zip` means the next push to the same PR overwrites it.
4. **Comment with preview link** builds `https://console.limrun.com/preview?asset=<name>&platform=ios` and posts it as a PR comment. Anyone with access to the PR can open it; the Limrun console provisions a simulator with the uploaded build pre-installed. The asset must already exist from step 3's `--upload`.

## Sign for real-device IPAs

If you want the preview to also produce a signed IPA (for TestFlight-like distribution outside Limrun's hosted simulator), add `--sdk iphoneos` and the signing flags. Base64-decode the cert and profile from secrets first:

```yaml
      - name: Build signed IPA on Limrun
        env:
          LIM_API_KEY: ${{ secrets.LIM_API_KEY }}
          P12_PASSWORD: ${{ secrets.P12_PASSWORD }}
        run: |
          echo "${{ secrets.P12_BASE64 }}"       | base64 -d > /tmp/dist.p12
          echo "${{ secrets.PROFILE_BASE64 }}"   | base64 -d > /tmp/MyApp.mobileprovision

          lim xcode build . \
            --scheme MyApp \
            --sdk iphoneos \
            --certificate-p12 /tmp/dist.p12 \
            --certificate-password "$P12_PASSWORD" \
            --provisioning-profile /tmp/MyApp.mobileprovision \
            --upload my-app-pr-${{ github.event.number }}.ipa
```

The build artifact lands in Asset Storage as `my-app-pr-N.ipa`. Fetch a signed download URL with `lim asset list --name my-app-pr-N.ipa --download-url` and post that alongside the preview link.

## Clean up when a PR closes

Each `lim xcode create` is cheap, but instances still count against your usage. Add a cleanup job that deletes everything tagged with the closed PR:

```yaml
  cleanup:
    if: github.event.action == 'closed'
    runs-on: ubuntu-latest
    steps:
      - run: npm install --global @limrun/cli
      - name: Delete PR instances and assets
        env:
          LIM_API_KEY: ${{ secrets.LIM_API_KEY }}
          PR: ${{ github.event.number }}
          REPO: ${{ github.event.repository.name }}
        run: |
          # Delete Xcode sandboxes for this PR
          lim xcode list --label-selector "pr=${PR},repo=${REPO}" --json \
            | jq -r '.[].metadata.id' \
            | xargs -r -n1 lim xcode delete

          # Delete the preview asset
          ASSET_NAME="${REPO}-pr-${PR}.zip"
          ASSET_ID=$(lim asset list --name "${ASSET_NAME}" --json | jq -r '.[0].id // empty')
          if [ -n "$ASSET_ID" ]; then
            lim asset delete "$ASSET_ID"
          fi
```

Trigger it from the same workflow file by adding `closed` to the `on.pull_request.types` array.

## Use the packaged Action for less boilerplate

If your CI is GitHub Actions, [`limrun-inc/ios-preview-action`](https://github.com/limrun-inc/ios-preview-action) wraps the create, build, upload, and PR-comment steps into one:

```yaml .github/workflows/ios-preview.yml
on:
  pull_request:
    types: [opened, synchronize, reopened, closed]

permissions:
  contents: read
  pull-requests: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: limrun-inc/ios-preview-action@main
        with:
          project-path: .
          api-key: ${{ secrets.LIM_API_KEY }}
```

Inputs: `project-path`, `project`, `workspace`, `scheme`, `sdk`, `api-key`, `github-token`. Outputs: `preview-url`, `asset-id`, `asset-name`.

Write the workflow by hand (as shown above) when you need finer control: a custom comment template, a different artifact-naming scheme, signing flags, or a runner outside GitHub Actions. The CLI is plain Node, so the same `lim xcode build --upload` pattern works on GitLab CI, CircleCI, Buildkite, Jenkins, and any other runner that can install `npm i -g @limrun/cli`.
