# Build with remote Xcode

Limrun's Xcode sandbox is a managed Mac build environment exposed over HTTPS. The loop is straightforward:

1. **Sync** your local source tree to the sandbox.
2. **Run `xcodebuild`** on the remote machine.
3. **Stream** stdout/stderr back over SSE while the build runs.
4. On success: auto-install on an attached simulator, upload to Asset Storage, or both.

**SDK coverage by language:**

| Surface | TypeScript | Python | Go | Java | CLI |
|---|---|---|---|---|---|
| Provision an Xcode instance | ✓ | ✓ | via REST (no `XcodeInstances`) | via REST (no `xcodeInstances()`) | ✓ |
| Sync source + run `xcodebuild` | ✓ (`xcodeInstances.createClient`) | not in SDK | not in SDK | not in SDK | ✓ (`lim xcode build`) |

Python, Go, and Java users provision the instance with their SDK, then shell out to `lim xcode build` for the actual sync and build. See [SDK capability matrix](/docs/reference#sdk-capability-matrix).

## Topology: standalone vs paired with iOS

There are two shapes for the Xcode sandbox, and the right one depends on whether you also need a simulator.

### Paired iOS + Xcode (most common)

When you want to build an app *and* run it on a simulator, create one iOS instance with the Xcode sandbox attached. This is what `lim ios create --xcode` and `spec.sandbox.xcode.enabled: true` do. A successful `xcodebuild` auto-installs and auto-launches the app on the simulator.

<CodeGroup>
```ts TypeScript
import Limrun from '@limrun/api';

const lim = new Limrun({ apiKey: process.env['LIM_API_KEY'] });

const instance = await lim.iosInstances.create({
  wait: true,
  reuseIfExists: true,
  metadata: { labels: { session: 'demo' } },
  spec: {
    sandbox: { xcode: { enabled: true } },
  },
});

const xcodeUrl = instance.status.sandbox!.xcode!.url!;
const xcode = await lim.xcodeInstances.createClient({
  apiUrl: xcodeUrl,
  token: instance.status.token,
});
```

```python Python
# Python provisions the paired instance; the build step itself is not in the
# Python SDK today. Drive the build with `lim xcode build` from your CI step.
instance = client.ios_instances.create(
    wait=True,
    reuse_if_exists=True,
    metadata={"labels": {"session": "demo"}},
    spec={"sandbox": {"xcode": {"enabled": True}}},
)
xcode_url = instance.status.sandbox.xcode.url
```

```go Go
// Go provisions the paired instance; the build step itself is not in the
// Go SDK today. Drive the build with `lim xcode build` from your CI step.
instance, err := lim.IosInstances.New(ctx, limrun.IosInstanceNewParams{
    Wait:          limrun.Bool(true),
    ReuseIfExists: limrun.Bool(true),
    Metadata: limrun.IosInstanceNewParamsMetadata{
        Labels: map[string]string{"session": "demo"},
    },
    Spec: limrun.IosInstanceNewParamsSpec{
        Sandbox: limrun.IosInstanceNewParamsSpecSandbox{
            Xcode: limrun.IosInstanceNewParamsSpecSandboxXcode{
                Enabled: limrun.Bool(true),
            },
        },
    },
})
xcodeURL := instance.Status.Sandbox.Xcode.URL
```

```java Java
// Java provisions the paired instance; the build step itself is not in
// the Java SDK today. Drive the build with `lim xcode build` from your CI.
IosInstance instance = client.iosInstances().create(
    IosInstanceCreateParams.builder()
        .wait(true)
        .reuseIfExists(true)
        .metadata(IosInstanceCreateParams.Metadata.builder()
            .labels(IosInstanceCreateParams.Metadata.Labels.builder()
                .putAdditionalProperty("session", "demo")
                .build())
            .build())
        .spec(IosInstanceCreateParams.Spec.builder()
            .sandbox(IosInstanceCreateParams.Spec.Sandbox.builder()
                .xcode(IosInstanceCreateParams.Spec.Sandbox.Xcode.builder()
                    .enabled(true)
                    .build())
                .build())
            .build())
        .build()
);
String xcodeUrl = instance.status().sandbox().get().xcode().get().url().get();
```

```bash CLI
lim ios create --xcode --reuse-if-exists --label session=demo
```
</CodeGroup>

The Xcode sandbox URL comes back at `instance.status.sandbox.xcode.url`. The same `token` as the iOS instance authenticates it.

### Standalone Xcode (no simulator)

For pure CI builds, real-device IPA distribution, or MCP-wrapped build tools, create a top-level Xcode instance. It has no simulator attached.

<CodeGroup>
```ts TypeScript
const xcodeInstance = await lim.xcodeInstances.create({
  wait: true,
  reuseIfExists: true,
  metadata: { labels: { session: 'ci-build' } },
});

const xcode = await lim.xcodeInstances.createClient({ instance: xcodeInstance });
```

```python Python
xcode_instance = client.xcode_instances.create(
    wait=True,
    reuse_if_exists=True,
    metadata={"labels": {"session": "ci-build"}},
)
# Drive the build with `lim xcode build` against this instance's ID.
```

```bash CLI
lim xcode create --reuse-if-exists --label session=ci-build
```
</CodeGroup>

The **Go and Java SDKs do not ship an `XcodeInstances` resource today.** From those languages, either provision an iOS instance with `sandbox.xcode.enabled: true` (the paired pattern above), POST to `/v1/xcode_instances` directly via HTTP, or call `lim xcode create` from your process.

You can also attach an existing simulator to a standalone Xcode sandbox after the fact:

```bash
lim xcode attach-simulator <ios_instance_id>
```

This lets the next build auto-install on that simulator.

## Sync your source

The CLI and `XcodeClient.sync()` both perform delta-based source sync. The first sync uploads everything; subsequent syncs upload only changed bytes using xdelta3 patches (with a configurable fallback to full uploads when patches exceed the budget).

<CodeGroup>
```ts TypeScript
await xcode.sync('./my-app', {
  watch: true,             // re-sync on file changes
  install: true,           // install after each sync (paired iOS only)
  maxPatchBytes: 4 * 1024 * 1024,
  basisCacheDir: '/tmp/lim-cache',
  additionalFiles: [
    { localPath: '/home/dev/.netrc', remotePath: '~/.netrc' },
  ],
});
```

```bash CLI
# One-shot sync followed by build (the most common path)
lim xcode build .

# Continuous sync (watch mode) without building
lim xcode sync . --watch
```
</CodeGroup>

### What gets synced

The sync always ignores `.git`, `.DS_Store`, the basis cache directory, and any path matched by your project's root `.gitignore` (case-sensitive). It also force-includes `.xcconfig` files even if a `.gitignore` rule would match them, because Xcode build configs need to be present on the remote.

Standard iOS build artifacts are excluded by the SDK on top of `.gitignore`:

- `build/`, `.build/`, `DerivedData/`, `Index.noindex/`, `ModuleCache.noindex/`, `.index-build/`
- `.swiftpm/`, `Pods/`, `Carthage/Build/`
- Anything containing `/xcuserdata/`
- Anything containing `.dSYM/`

If a file is both tracked by git and ignored, the SDK emits a warning so you can fix the mismatch.

### Additional files

If your build needs files that aren't in your repo (a `.netrc` for private package access, a CI-only `Config.swift`, a secrets file), use `additionalFiles` on the SDK or `--additional-file local=remote` on the CLI. Paths starting with `~/` are expanded to the sandbox's home directory.

### Tuning the patch budget

`maxPatchBytes` (default `4 * 1024 * 1024`, or 4 MiB) caps the size of any single xdelta3 patch. When a patch would exceed it, the SDK falls back to a full upload for that file. Lower it on slow networks; raise it for large source changes.

`basisCacheDir` holds the last-synced copy of every file so the SDK can compute patches locally without round-tripping the server. Defaults to a path under the OS temp dir, scoped by the sync target. Pinning it (e.g. `/var/cache/lim`) keeps the cache warm across CI runs.

## Run `xcodebuild`

<CodeGroup>
```ts TypeScript
const build = xcode.xcodebuild({
  workspace: 'MyApp.xcworkspace',
  scheme: 'MyApp',
  sdk: 'iphonesimulator', // or 'iphoneos', 'watchsimulator', 'watchos'
});

build.command.on('data', (line) => process.stdout.write(line));
build.stdout.on('data', (line) => process.stdout.write(line));
build.stderr.on('data', (line) => process.stderr.write(line));

const { exitCode, status } = await build;
// status: 'SUCCEEDED' | 'FAILED' | 'CANCELLED'
```

```bash CLI
lim xcode build . --scheme MyApp --workspace MyApp.xcworkspace
```
</CodeGroup>

### Build streams

`xcodebuild` returns a `ExecChildProcess` with three `ReadableStream` channels and a final `ExecResult` when you await it:

| Channel | Carries |
|---|---|
| `command` | The fully-expanded command string the sandbox executed. One event, then closes. |
| `stdout` | xcodebuild stdout as the build runs. |
| `stderr` | xcodebuild stderr. |

The awaitable result has `exitCode`, `status` (`SUCCEEDED` / `FAILED` / `CANCELLED`), and (when `upload.assetName` was passed) a `signedDownloadUrl` you can hand to a teammate or a reviewer.

### Project, workspace, scheme, SDK

Match what you'd pass to local `xcodebuild`:

| Field | Use |
|---|---|
| `workspace` | Path to a `.xcworkspace`, e.g. `MyApp.xcworkspace`. |
| `project` | Path to a `.xcodeproj`. Use this *or* `workspace`, not both. |
| `scheme` | Required for multi-scheme apps. |
| `sdk` | `iphonesimulator` (default for the simulator), `iphoneos` for device builds, `watchsimulator` / `watchos` for watchOS. |

## Code signing for real devices

To produce a signed IPA for distribution, pass a P12 certificate and a provisioning profile. The SDK base64-encodes them before sending; the CLI handles encoding for you.

<CodeGroup>
```ts TypeScript
import fs from 'node:fs';

const build = xcode.xcodebuild(
  { scheme: 'MyApp', sdk: 'iphoneos' },
  {
    signing: {
      certificateP12Base64: fs.readFileSync('./dist.p12').toString('base64'),
      certificatePassword: process.env['P12_PASSWORD'],
      provisioningProfileBase64: fs.readFileSync('./MyApp.mobileprovision').toString('base64'),
    },
    upload: { assetName: 'my-app-pr-42.ipa' },
  },
);

const { exitCode, signedDownloadUrl } = await build;
```

```bash CLI
lim xcode build . \
  --sdk iphoneos \
  --scheme MyApp \
  --certificate-p12 ./signing/dist.p12 \
  --certificate-password "$P12_PASSWORD" \
  --provisioning-profile ./signing/MyApp.mobileprovision \
  --upload my-app-pr-42.ipa
```
</CodeGroup>

`signedDownloadUrl` is returned on the build result when you pass `upload`. Store the URL in your PR comment or pipeline output. Anyone with the URL can download the IPA without a Limrun API key (the signature expires after a short window).

## Upload the artifact

There are two ways to capture a build output for distribution.

**Upload to Asset Storage** with `upload: { assetName: '...' }` (or `--upload <name>` on the CLI). The artifact lives in Limrun's managed storage and is referenceable by name in subsequent `lim ios create --install-asset <name>` calls. Asset Storage is also what backs the [PR Previews](/docs/ios/pr-previews) flow.

**Upload to your own bucket** with `upload: { signedUploadUrl: '...' }` (or `--signed-upload-url <url>`). Provide a pre-signed S3 / GCS / R2 URL and the artifact is PUT to it directly from the sandbox. Use this when you want the artifact to land in storage you already manage.

## Auto-install on the simulator

When the Xcode sandbox is paired with an iOS instance (the `lim ios create --xcode` path, or after `lim xcode attach-simulator`), every successful build auto-installs and auto-launches the app on that simulator. There's no extra step.

For standalone Xcode instances, attach a simulator at any time:

```bash
lim xcode attach-simulator <ios_instance_id>
```

Or via SDK:

```ts
await xcode.attachSimulator(iosInstance);
// or with raw URL/token:
await xcode.attachSimulator({ apiUrl: '...', token: '...' });
```

## Cookbook: end-to-end CI build

The smallest practical Linux CI job that produces a signed IPA and posts a download link:

```yaml .github/workflows/build.yml
name: iOS build
on: { pull_request: { types: [opened, synchronize] } }
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install --global @limrun/cli
      - name: Build
        env:
          LIM_API_KEY: ${{ secrets.LIM_API_KEY }}
          P12_PASSWORD: ${{ secrets.P12_PASSWORD }}
        run: |
          echo "${{ secrets.P12_BASE64 }}" | base64 -d > /tmp/dist.p12
          echo "${{ secrets.PROFILE_BASE64 }}" | base64 -d > /tmp/MyApp.mobileprovision
          lim xcode create --reuse-if-exists --label pr=${{ github.event.number }}
          lim xcode build . \
            --scheme MyApp \
            --sdk iphoneos \
            --certificate-p12 /tmp/dist.p12 \
            --certificate-password "$P12_PASSWORD" \
            --provisioning-profile /tmp/MyApp.mobileprovision \
            --upload my-app-pr-${{ github.event.number }}.ipa
```

No `runs-on: macos-latest`. No Apple-licensed CI runner. The build happens on Limrun's Mac fleet; the runner only needs to call `lim`.
