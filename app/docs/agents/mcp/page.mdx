# MCP for controlling instances

Every iOS and Android instance Limrun creates has its own Model Context Protocol (MCP) server. Three things to know:

- **One server per instance, not one shared endpoint.** The URL is in `instance.status.mcpUrl`.
- **Auth uses your org API key**, not `instance.status.token`. Pass it as `Authorization: Bearer lim_…`.
- **Five tools per iOS instance.** A combined screenshot + element-tree tool, one batch tool (`ios-use`) that carries every UI action, plus dedicated tools for app lifecycle, logs, and deep links. See [What the MCP server exposes](#what-the-mcp-server-exposes) below.

This page covers when to use MCP instead of the CLI, how to find the URL, and how to wire it into Claude Code, Claude Desktop, Cursor, Codex, and custom MCP clients.

## When to use the CLI, MCP, or both

| Aspect | `lim` CLI | Per-instance MCP server |
|---|---|---|
| **Where it runs** | The agent's shell | The agent's MCP client transport |
| **Auth** | Org API key in env or config | Same org API key as `Authorization: Bearer` (not `status.token`) |
| **Connection** | Fresh WebSocket per command (or a persistent session via `lim session start`) | Long-lived MCP session |
| **What it can do** | Every `lim ios *`, `lim xcode *`, `lim asset *`, and `lim android *` command | The device-control subset only: no `xcode build`, no `lim asset push` |
| **Best for** | Builds, instance lifecycle, asset management | Fast tap-screenshot-elementTree loops where every round-trip matters |

Most agents end up using both. Use the CLI for provisioning, builds, and assets. Use MCP when the agent needs native tool-use over the device-control surface without shelling out.

## Get the `mcpUrl`

`mcpUrl` is returned in `instance.status` once the instance reaches `ready`. The URL itself has no credentials in it. Auth goes in an `Authorization: Bearer` header, and the bearer is your **org API key** (the same `lim_...` key you use with the REST API), not the per-instance `instance.status.token`.

<CodeGroup>
```ts TypeScript
import Limrun from '@limrun/api';

const lim = new Limrun({ apiKey: process.env['LIM_API_KEY'] });

const instance = await lim.iosInstances.create({
  wait: true,
  reuseIfExists: true,
  metadata: { labels: { name: 'agent-session' } },
});

console.log(instance.status.mcpUrl);
// e.g. https://<dc>-<shard>.limrun.net/v1/ios_<id>/mcp
```

```python Python
from limrun_api import Limrun

client = Limrun()
instance = client.ios_instances.create(wait=True, reuse_if_exists=True)
print(instance.status.mcp_url)
```

```go Go
client := limrun.NewClient()
inst, _ := client.IosInstances.New(ctx, limrun.IosInstanceNewParams{
    Wait:          limrun.Bool(true),
    ReuseIfExists: limrun.Bool(true),
})
fmt.Println(inst.Status.McpURL)
```

```java Java
// The Java SDK does not yet expose `mcpUrl` on the typed Status model.
// Read it from the raw additional-properties map (or use `lim ios get --json`).
IosInstance instance = client.iosInstances().create(
    IosInstanceCreateParams.builder()
        .wait(true)
        .reuseIfExists(true)
        .metadata(IosInstanceCreateParams.Metadata.builder()
            .labels(IosInstanceCreateParams.Metadata.Labels.builder()
                .putAdditionalProperty("name", JsonValue.from("agent-session"))
                .build())
            .build())
        .build()
);
JsonValue mcpUrl = instance.status()._additionalProperties().get("mcpUrl");
```

```bash CLI
lim ios get --json | jq -r '.status.mcpUrl'
```
</CodeGroup>

## Wire it into your agent

### Claude Code

The command from the Connect modal:

```bash
claude mcp add simulator --transport http \
  '<mcpUrl>' \
  --header 'Authorization: Bearer lim_<your_api_key>'
```

Replace `<mcpUrl>` with the value from `instance.status.mcpUrl` and `lim_<your_api_key>` with your Limrun API key. Verify with `claude mcp list`.

The console renders the same command with your real values filled in, ready to copy:

<Frame>
  <img src="/images/console/claude-code-limrun.png" alt="Connect modal on console.limrun.com with the Claude Code section expanded, showing the full claude mcp add simulator --transport http <mcpUrl> --header 'Authorization: Bearer lim_...' command ready to copy" />
</Frame>

### Claude Desktop

Add the URL to your MCP servers config in Claude Desktop's settings:

```json
{
  "mcpServers": {
    "limrun-ios": {
      "type": "http",
      "url": "<mcpUrl>",
      "headers": {
        "Authorization": "Bearer lim_<your_api_key>"
      }
    }
  }
}
```

### Cursor

The console's "Add to Cursor" button opens Cursor with the MCP server pre-filled. To set it up by hand, add a new MCP server in Cursor settings with `<mcpUrl>` and the `Authorization: Bearer lim_<your_api_key>` header.

### Codex Web or any custom MCP client

Any MCP client with HTTP transport works. Pass the `mcpUrl` as the endpoint and `Authorization: Bearer lim_<your_api_key>` as a header. The Connect modal's **Custom** section gives you both values directly:

## What the MCP server exposes

Each iOS instance exposes five MCP tools. The bulk of the device-control surface lives in a single batch tool (`ios-use`); the others handle the read-state loop, app management, logs, and deep links.

| Tool | What it does |
|---|---|
| `ios-screenshot-and-element-tree` | Returns the current screenshot and the accessibility tree in one call. The agent calls this before each `ios-use` batch to know what's on the screen. |
| `ios-use` | Runs an ordered list of UI actions: tap, type, scroll, recording, orientation, multi-touch, StoreKit testing, soft reset, and more. See the action kinds below. |
| `ios-app` | List, launch, or terminate installed apps. (Does not install apps; use the CLI or per-instance REST for that.) |
| `ios-open-url` | Open a URL or a deep link. Preferred over the `openUrl` action inside `ios-use`. |
| `ios-logs` | Tail combined stdout/stderr for an installed app's bundle ID. |

### `ios-use` action kinds

A single `ios-use` call carries an ordered `actions` array. The `kind` field on each item selects the action:

- **Tap and select**: `tap`, `tapElement`, `incrementElement`, `decrementElement`, `setElementValue`
- **Input**: `type`, `pressKey`, `toggleKeyboard`
- **Movement**: `scroll`, `setOrientation`
- **Low-level HID**: `touchDown`, `touchMove`, `touchUp`, `keyDown`, `keyUp`, `buttonDown`, `buttonUp`
- **Recording**: `startRecording`, `stopRecording`
- **State and reset**: `deviceInfo`, `wait`, `softReset`
- **StoreKit testing**: `discoverStoreKitConfig`, `clearStoreKitConfig`
- **Navigation**: `openUrl` (prefer the `ios-open-url` tool above)

For action parameters (selector fields, coordinates, durations) and the full per-method reference, see [Run an iOS Simulator](/docs/ios/run-simulator). For the Android equivalent, see [Run an Android Emulator](/docs/android/run-emulator).

## Build your own MCP server around `lim`

Sometimes the hosted MCP surface isn't quite what you want. Maybe you want `xcode build` exposed as a tool, or you want to wrap several `lim` commands into one higher-level tool. Building your own MCP server is straightforward: wrap the SDK or the CLI in whatever shape you need.

The [`ios-with-xcode` example](https://github.com/limrun-inc/typescript-sdk/tree/main/examples/ios-with-xcode) is a working pattern: it uses `@modelcontextprotocol/sdk` to expose `xcode.xcodebuild()` as a single `build` MCP tool, then mounts it at `http://localhost:3000` for the agent to use.

```ts
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StreamableHTTPServerTransport } from '@modelcontextprotocol/sdk/server/streamableHttp.js';
import Limrun from '@limrun/api';

const lim = new Limrun({ apiKey: process.env['LIM_API_KEY'] });
const instance = await lim.iosInstances.create({
  wait: true,
  reuseIfExists: true,
  spec: { sandbox: { xcode: { enabled: true } } },
});
const xcode = await lim.xcodeInstances.createClient({
  apiUrl: instance.status.sandbox!.xcode!.url!,
  token: instance.status.token,
});

const mcp = new McpServer({ name: 'xcodebuild', version: '1.0.0' });
mcp.registerTool(
  'build',
  {
    title: 'xcodebuild',
    description: 'Run xcodebuild on the remote Xcode sandbox and install on the simulator.',
  },
  async () => {
    const build = xcode.xcodebuild();
    const chunks: string[] = [];
    build.stdout.on('data', (line) => chunks.push(line.toString()));
    await build;
    return { content: [{ type: 'text', text: chunks.join('\n') }] };
  },
);
```
